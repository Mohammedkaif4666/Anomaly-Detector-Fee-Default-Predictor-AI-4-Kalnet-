## ⚙️ 1. Backend Architecture (`jyothsna_lenka/main.py`)

All backend logic is consolidated inside `jyothsna_lenka/main.py` using **FastAPI**.

### A. Asynchronous Startup Cache (The Loading Strategy)
Normally, running machine learning models on a database takes time (seconds). If the server did this every time someone clicked a button, the page would freeze.
- **What Jyothsna Did:** She wrote a startup function (`@app.on_event("startup")`) that runs exactly **once** when the server starts.
- **How it works:**
  1. It loads the models (`model.pkl`, `scaler.pkl`) from the disk into RAM.
  2. It reads the CSV datasets (`attendance_features.csv`, `fee_features.csv`, `student_labels.csv`).
  3. It runs prediction inference for **all 500 students** immediately.
  4. It stores the results in a global Python list called `students_cache`.
- **Why this matters:** When a user goes to any page, the data loads in **less than 10 milliseconds** because the calculations are already pre-computed and stored in memory.

### B. Route Definitions (Page Serving)
FastAPI acts as the web host. Jyothsna mapped the page URLs to serve their corresponding HTML files:
- `/` serves [landing.html](file:///c:/Users/moham/Downloads/kalnet%20AI-4%20team%20project/templates/landing.html) (Public home page).
- `/students` serves [index.html](file:///c:/Users/moham/Downloads/kalnet%20AI-4%20team%20project/templates/index.html) (Public student directory table).
- `/dashboard` serves [dashboard.html](file:///c:/Users/moham/Downloads/kalnet%20AI-4%20team%20project/templates/dashboard.html) (Secure admin analytics dashboard).
- `/docs` serves the built-in interactive **Swagger API documentation**.

### C. Admin Security (Basic Auth)

## 📡 2. API Endpoints Exposed (The Connection Layer)

The frontend JavaScript communicates with the backend by making **HTTP requests** (fetching JSON). Jyothsna created the following REST API endpoints:

### 1. `GET /api/summary`
- **What it does:** Returns simple totals for cards (total students, flagged anomalies, fee defaults).
- **Used by:** Landing page metrics, Student Directory header stats, Admin Dashboard KPI cards.

### 2. `GET /api/students`
- **Parameters:** `search` (text), `filter` (`all`, `anomaly`, `fee_risk`, `safe`).
- **What it does:** Searches through the RAM cache matching student names or IDs, filters them by their ML risk classification, and returns the list.
- **Used by:** The student table on `/students`.

### 3. `GET /api/student/{student_id}`
- **What it does:** Fetches the specific student details from `students_cache`, reads the raw file `attendance_raw.csv`, retrieves their last 30 daily presence/absence records, and returns it.
- **Used by:** The popup modal when you click "View" on any student to render their 30-day bar chart.

### 4. `GET /api/analytics`
- **What it does:** Aggregates cache statistics for the charts:
  - Attendance rates grouped in 10% buckets.
  - Risk distribution classifications (High >70, Medium 40-70, Low <40).
  - Class-wise section groupings (counting defaults, late, anomalies per class: e.g. 6A, 8B).
  - A pre-sorted list of all students ranked by combined risk (Attendance risk + Fee probability default).
- **Used by:** The Admin Dashboard visualizations.

### 5. `POST /ai/anomalies` & `POST /ai/fee-risk`
- **What it does:** Allows external programmatic access. Takes a JSON list of student IDs, runs the raw ML models, and returns their immediate scores.

## 🖥️ 3. Frontend Interactivity & Controls (Admin Panel)

Jyothsna integrated interactive features in [dashboard.html](file:///c:/Users/moham/Downloads/kalnet%20AI-4%20team%20project/templates/dashboard.html) that communicate directly with her APIs.

### A. Chart segment Click Actions (Chart.js Interactions)
Every chart component in the Admin Dashboard is clickable:
1. **Donut Slices (Risk / Fees):** Clicking a slice (e.g. "High Risk" or "Default") filters `allStudents` in JavaScript memory and opens a popup displaying those exact students.
2. **Attendance Bars:** Clicking a bar (e.g., "50-60%") lists the students falling inside that specific attendance bucket.
3. **Class Bars:** Clicking a grouped bar (e.g., anomalies in Class "9A") extracts the exact students in class 9A who have been flagged.

### B. Staggered Double Modals (Z-Indexing)
- **The Challenge:** How can the admin inspect a student's profile without losing their place inside a chart segment list popup?
- **Jyothsna's Solution:** She built a layered modal interface. The segment popup table has a z-index of `500`. Clicking "View" on a student row opens the Single Student Profile card with a z-index of `600` directly on top of the list.
- **User Experience:** Closing the profile modal retains the segment list underneath, letting the admin continue auditing.

### C. Top N limit selector
- **How it works:** A select element in the "Highest-Risk Students" header toggles options between Top 10, 20, 30, 50, or Show All.
- **Implementation:** JavaScript slices the pre-sorted array (`topRiskStudents.slice(0, N)`) and redraws the table rows instantly.

### D. Autocomplete Navbar Search
- **How it works:** A quick-lookup bar resides in the top nav. On typing (`oninput`), a suggestion box drops down displaying names, IDs, and colored badges.
- **UX detail:** She used `onmousedown` for suggestions click to prevent the input focus-blur (`onblur`) from hiding the suggestions list before a click registers.
